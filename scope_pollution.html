Scope Pollution

It may seem like a great idea to always make your variables accessible, but having too many global variables can cause problems in a program.

When you declare global variables, they go to the global namespace. The global namespace allows the variables to be accessible from anywhere in the program. These variables remain there until the program finishes which means our global namespace can fill up really quickly and it becomes difficult to keep track of what variable is where. This can lead to bugs that are hard to find because we don't know where something came from.

Scope pollution is a problem in JavaScript where variables that are declared inside of functions have the same scope as global variables. This can lead to unexpected behavior if not handled properly.

Scope pollution makes it difficult to keep track of our different variables and sets us up for potential accidents. For example, globally scoped variables can collide with other variables that are more locally scoped, causing unexpected behavior in our code. 

            // Global Variable
            let x = "I am global";
            function myFunction() {
                let x = "I am local";
                console.log(x); // Output: I am local
                }
                myFunction();
                console.log(x); // Output: I am global
In the example above, you will notice the following:
  • We have two variables named 'x'. One is global (outside any function) and one is local (inside the function).
  • Inside the function, when we log `console.log(x)`, it logs `"I am local"` instead of `"I am global"`. This demonstrates how scope works.
  • Outside the function, logging `console.log(x)` logs `"I am global"`, showing that even though there was a local variable called 'x', the global variable was used.
NOTE:While it’s important to know what global scope is, it’s best practice to not define variables in the global scope.
